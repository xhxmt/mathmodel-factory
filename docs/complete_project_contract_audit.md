# Complete Project Contract Audit

This document records the current delivery contract for completed Modeling
Factory projects and how legacy projects are classified after the July 2026
quality-gate tightening.

Generated contract status is stored in `complete/_validation_index.json`.
Each audited project also has `complete/<base>/delivery_manifest.json` when the
audit is run with `--write-manifests`.

## Status Meanings

- `CURRENT_PASS`: satisfies the current delivery contract.
- `LEGACY_DELIVERED`: has delivered PDF/zip artifacts, but fails at least one current gate.
- `INVALID_OR_INCOMPLETE`: missing delivery artifacts or too incomplete to treat as delivered.

## Current Contract

The active contract version is defined in `scripts/delivery_contract.py`:

```text
2026-07-02.step8_5_gate2_zip
```

A project can be `CURRENT_PASS` only when all of the structural checks in
`scripts/evaluate_modeling_project.py` pass and `run_paper.sh --infer-step`
reports Step 16. The practical hard gates are:

- Step 8.5 reviewer-entry artifacts exist and `entry_gate.md` resolves to `PASS`.
- Gate 2 judge verdict is exactly `VERDICT: PASS`.
- `numbers_manifest.json` exists and `verify_numbers.py --verify` passes.
- Symbol audit passes, or the allowed degraded-symbol path is explicitly documented in `code_review.md`.
- Canonical result files under `results/**` do not report incomplete states such as `RUNNING`, `PARTIAL`, or `FAILED`.
- Project PDF, copied `papers/<base>_paper.pdf`, and `papers/<base>_submission.zip` exist, and the zip is readable.

The shared predicates for Gate 2, Step 8.5, zip validity, and Step 16 readiness
now live in `scripts/workflow_state.py`. Runner and evaluator code should call
that helper instead of adding new ad hoc shell parsing.

## Implemented Changes

### Delivery Manifest

`run_paper.sh` writes `delivery_manifest.json` only after the final delivery
quality gate passes. The manifest is generated by `scripts/delivery_contract.py`
and records:

- contract version;
- generated timestamp;
- `CURRENT_PASS` / `LEGACY_DELIVERED` / `INVALID_OR_INCOMPLETE` status;
- runner commit, when available;
- inferred step and failed non-warning checks;
- PDF and submission zip size/hash records.

If manifest generation fails, Step 16 fails and the project remains incomplete.

### Complete Directory Audit

`scripts/audit_complete_projects.py` audits all projects under `complete/`.
With `--write-manifests`, it writes:

- `complete/_validation_index.json`;
- `complete/<base>/delivery_manifest.json` for every audited project.

This is intentionally non-destructive. Existing delivered projects remain in
`complete/`; the status label tells whether they satisfy the current contract.

### Evaluation Result Enrichment

`evaluation/run_evaluation.sh` now keeps the structural precheck JSON at
`evaluation/results/<base>_precheck.json` and enriches the aggregate judge
output through `scripts/enrich_evaluation_result.py`.

The enriched `<base>_eval.json` contains:

- `structural`: hard evidence such as `precheck_passed`, inferred step,
  unmatched-number count, and blocking checks;
- `llm_score`: median/min/max/spread scores and verdict distribution;
- `comparison_ready`: true only when structural precheck passes and at least one
  LLM score sample parsed.

This separates "the project is structurally deliverable" from "an external LLM
liked the paper." Comparisons should not be based on a single total score alone.

### Model Dispatch Config Helper

`scripts/model_dispatch_config.py` centralizes parsing of
`web/model_config.json` and `web/model_registry.json`. `run_paper.sh` now calls
this helper for step model assignments and registry entries instead of embedding
inline Python.

Future runner changes should follow this pattern: put reusable parsing/state
logic in small Python helpers, then keep shell code as orchestration glue.

### Runtime Secrets

`launch_agents.sh` no longer loads `scripts/load_secrets.sh` at launcher start.
Runtime secret loading is handled inside `run_paper.sh`, so project listing and
launcher commands do not fail merely because cloud secret loading is unavailable.

## Commands

Audit all completed projects and write manifests:

```bash
python3 scripts/audit_complete_projects.py --write-manifests
```

Audit without writing files:

```bash
python3 scripts/audit_complete_projects.py --no-write
```

Write a manifest for one project:

```bash
python3 scripts/delivery_contract.py complete/<base> --root "$(pwd)"
```

Inspect workflow gate state for one project:

```bash
python3 scripts/workflow_state.py json complete/<base> --root "$(pwd)"
```

Run external evaluation with separated structural and LLM evidence:

```bash
./evaluation/run_evaluation.sh complete/<base> --samples 3
```

## Reclassification Rule

Do not manually edit `_validation_index.json` or `delivery_manifest.json` to
promote a project. A project should move from `LEGACY_DELIVERED` to
`CURRENT_PASS` only after its source artifacts are repaired and the audit command
regenerates a passing manifest.
