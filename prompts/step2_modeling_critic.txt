[Verdict 体系]
本步骤定义**三种** verdict（与原社科版不同，请严格区分）：

```
VERDICT: VALIDATED
```
含义：本流可以推进到 Step 3 决策。允许有正常局限（如假设强、demo 规模小），但**没有立即致命问题**。

```
VERDICT: REVISE
```
含义：有**可修复**的问题，proposal agent 再跑一轮能改好。runner 会把本流送回 step2_modeling_proposal 重做。

```
VERDICT: ABANDONED
```
含义：本流**结构性不可推进**——公式自相矛盾、demo 怎么改也跑不通、所选方法在该子问题集上根本不适用、数据缺口大到任何合理假设都补不上。本流被永久淘汰，**但 spec / demo 产物保留**作为 Step 3 比较和事后复盘的素材。判 ABANDONED 必须给出充分的、**形式化**的论据，不要"凭感觉"。

第一行**必须**精确匹配 `^VERDICT: (VALIDATED|REVISE|ABANDONED)$`（runner 用既有 `step2_critique_verdict()` 解析，多空格 / 句号 / 中文标点都会让 runner 卡死）。

[判 ABANDONED 的硬阈值]
满足**任意一条**才允许判 ABANDONED：
1. 公式被证明在题目原约束下**没有可行解**（不是 demo 数据巧合 infeasible，而是数学上 always infeasible，需要 spec 自带反例或推理）
2. 所选主方法的核心假设与 `terminology_table.md` 已锁定的术语定义直接冲突（spec 里没有"我们重新定义术语"的合法理由）
3. demo 跑了 ≥ 2 轮（critic 已判过 REVISE 一次），spec 改过但同类硬错仍存在（即"REVISE 用尽"——proposal agent 显然能力 / 信息不足以救活）
4. 主方法 `method_library/...md` 路径不存在（应在 Step 1 拦截，但 critic 是兜底）
5. spec 第一行已经是 `# ABORT: ...`（proposal agent 自报阵亡）

不满足以上任意一条但仍有问题 = **REVISE**，不是 ABANDONED。倾向于给 REVISE 一次机会而不是直接 abandon——Step 3 决策更喜欢多看几个候选。

[判 VALIDATED 的硬条件]
**全部**满足才允许判 VALIDATED：
1. `__STREAM_PREFIX___spec.md` 存在且 ≥ 30 行（runner 也会查，但你先查）
2. `__STREAM_PREFIX___demo_result.json` 存在、JSON 合法、`status` 字段为 OPTIMAL/COMPLETED/CONVERGED 等明确成功值
3. JSON 里的 `jobid` **确实存在**于 `__FACTORY__/run_state/solver_jobs/`（用 `ls __FACTORY__/run_state/solver_jobs/<jobid>.meta` 验证）—— 这是反伪造硬检查
4. `models/__STREAM_PREFIX___*/demo.py`（或同类）真实存在且非空
5. spec 中的方法 / 子问题覆盖范围**与 viable_streams.md 中本流授权一致**（没有偷偷换方法）
6. spec 的"集合/参数/决策变量"三个表里，所有出现在目标和约束里的符号都登记了（无未声明符号）
7. demo 的 sanity_checks 至少有 2 条且 passed=true，且 `value` 字段有具体数值（不是"true/None"占位）
8. spec 的"创新点"段不是空话（如"考虑了实际情况"是空话，"用 McCormick 线性化处理乘积约束 $x_i y_j$"是具体）

任一条不满足 ⇒ 不能 VALIDATED。其他立即致命问题的判定见下方。

[立即致命问题 → REVISE 或 ABANDONED]
- memo 与 demo 不一致：spec 声称目标值在 $10^5$ 量级，JSON 里却是 $10^{-3}$
- demo 数据规模与 spec 描述对不上（spec 说 n=50，JSON 写 n=5000——可能 demo 跑的不是 demo）
- 求解器调用没走 `solver_submit.sh`（log 里没有 `local_*` 或 `cloud_*` 形式的 jobid，或 `run_state/solver_jobs/` 里查无此号）
- 关键约束写错（如目标是最大化但写成 min、容量约束方向写反）—— 这种是 REVISE，proposal agent 能改
- 假设过强到把题目变了样：如题目要求"动态决策"，spec 用"静态一次性决策"绕开
- spec 直接抄 `method_library/<>.md` 的玩具数据没改成本题情境
- 求解器跑的不是 spec 公式：demo.py 里实际写的目标 / 约束与 spec 写的不一致（这是 demo 不可信）
