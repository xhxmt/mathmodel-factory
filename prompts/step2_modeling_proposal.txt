```markdown

## 关键假设

- **A1**：<假设陈述> —— 依据：<题目原文 / 团队判断 / 文献> —— 若违反：<可能的影响>

| 符号 | 含义 | 取值范围 |
```

### 2. `__PROJECT_PATH__/__STREAM_PREFIX___demo_result.json`（**必需，机器可读**）

调用 `__FACTORY__/solver_submit.sh` 跑一次**小规模真实求解**，把关键数值结果存到这个 JSON。这是 critic 用来判断"公式能跑"的硬证据。

JSON 必含字段（示例）：

```json
{
  "stream": "m__STREAM_ID__",
  "demo_script": "models/__STREAM_PREFIX___<short>/demo.py",
  "solver": "gurobipy 11.0.3",
  "jobid": "local_python_20260522_153021_4711",
  "runtime_sec": 18.4,
  "scale": {"n_vars": 50, "n_constraints": 30, "n_periods": 12},
  "status": "OPTIMAL",
  "objective": 12345.67,
  "key_decision_summary": {
    "<决策变量名>": <值 或 列表节选>,
    "...": "..."
  },
  "sanity_checks": [
    {"name": "约束 C1 不违反", "passed": true, "value": "<具体数值>"},
    {"name": "目标在合理量级", "passed": true, "value": "<具体数值>"}
  ],
  "notes": "<对结果合理性的一句话评论>"
}
```

可选附加产物：
- `__PROJECT_PATH__/__STREAM_PREFIX___demo_result.csv`：如果 demo 产生表格型输出（如多时刻决策、多场景 KPI）
- `__PROJECT_PATH__/logs/__STREAM_PREFIX___demo.log`：solver_submit.sh 写出的 stdout/stderr 副本（解算结束后必须 mv 进来）

求解器调用硬规则（违反 = critic 直接 REVISE）：
- **必须**用 `__FACTORY__/solver_submit.sh`，不要 `nohup python ... &`
- **必须**给 `--max-time`，demo 总耗时建议 ≤ 15 分钟（900 秒），单 job 通常 ≤ 5 分钟
- 跑完用 `--wait` 拿终态，把 status (`COMPLETED`/`FAILED`/`TIMEOUT`) 体现在 JSON 里
- demo 数据 ≠ 题目全规模数据。用 sub-sample 或人造小例子；demo 落在 `data/intermediate/__STREAM_PREFIX____demo/` 下，**不要污染** `data/raw/`

[执行流程]
1. 读 `viable_streams.md`，定位 `## Stream m__STREAM_ID__:` 段。把它**完整复制**进你的草稿（不许漏字段）。如本段缺失或与 `viability_gate.md` 里你的流状态矛盾，立即写 `__STREAM_PREFIX___spec.md` 第一行 `# ABORT: viable_streams.md 中未找到 Stream m__STREAM_ID__` 并退出（这是真实异常，runner 会捕捉到）。
2. 读 `viability_gate.md`，确认本流的甄别标记。若你的流有 ⚠️ 注记（如"需要外部数据"），在 spec 的"关键假设"段必须显式接住。
3. 读 `__FACTORY__/method_library/<your-method>.md`：抄它的"数学形式"和"代码模板"做起点，按本题情境改造；不要照搬模板的玩具数据。
4. 写 `__STREAM_PREFIX___spec.md` 草稿（先不写 demo 结果）。
5. 在 `models/__STREAM_PREFIX___<short>/` 写 `demo.py`：
   - 输入数据规模刻意小（如题目附件第一周的数据，或 N=50 的人造样本）
   - 求解器调用按 method_library 模板
   - **结果序列化到 JSON**（绝对路径写 `__PROJECT_PATH__/__STREAM_PREFIX___demo_result.json`）
6. 提交：`__FACTORY__/solver_submit.sh --type python --max-time 900 models/__STREAM_PREFIX___<short>/demo.py`，记下 JOBID。如有多个独立 demo（如多个 sub-problem），并行提交。
   - JOBID 可能以 `local_` 或 `cloud_` 开头；两者都必须能在 `__FACTORY__/run_state/solver_jobs/<jobid>.meta` 查到。
7. `--wait $JOBID`，看 status：
   - COMPLETED：读 `models/__STREAM_PREFIX___<short>/demo.log`，把关键数填回 JSON 的 `objective`、`runtime_sec`、`sanity_checks`
   - FAILED / TIMEOUT：先看是不是公式错（如 LP infeasible 是真问题），改公式或换数据规模重跑；**不要**把 FAILED 状态写进 JSON 就交差
8. 把 `<demo>.log` 移到 `logs/__STREAM_PREFIX___demo.log`（保持项目根整洁，per modeling_guide.md）
9. 回填 `__STREAM_PREFIX___spec.md` 的"Demo 设计说明"段，引用 JSON 里的关键数（如"demo 在 n=50 上 18s 求得目标值 12345.67"）
10. 自检：
    - `test -f __PROJECT_PATH__/__STREAM_PREFIX___spec.md && [ $(wc -l < ...) -ge 30 ]`
    - `test -s __PROJECT_PATH__/__STREAM_PREFIX___demo_result.json` 且 JSON 合法（`python -m json.tool` 不报错）
    - `test -d __PROJECT_PATH__/models/__STREAM_PREFIX___*` 且其中至少有 `demo.py` + `README.md`
    - `test -f __PROJECT_PATH__/logs/__STREAM_PREFIX___demo.log` 或同等位置（不强制——某些 solver 不写 log）
    - JSON 里 `status` 是 OPTIMAL/COMPLETED/CONVERGED 这一类成功值（不是 FAILED）
11. **不要**更新 `checkpoint.md`（Step 2 是并行多流；runner 等所有流 critic 通过后才更新）

[质量边界]
- **demo 必须真跑过**，不许只写脚本不提交。Critic 第一件事就是看 JSON 里的 jobid 在不在 `__FACTORY__/run_state/solver_jobs/` 下。
- **公式严格自洽**：所有出现在目标和约束里的符号必须在"集合/参数/决策变量"三个表里登记。Spec 里出现的随手符号是 Step 4 最容易爆雷的地方。
- **不要写完整模型**：Step 4 是全模型构建。本步骤只要"立项级 + 跑通最小例子"。Spec 里写"详细求解算法 / 灵敏度 / 完整子问题"是越界。
- **不要伪造结果**：JSON 里的数值必须来自实际的 solver log。如果 demo 跑不通，**走 critic-revise 循环**（写出真实状态，让 critic 判 REVISE），不要瞎填数。
- **不要互相依赖**：你的流不应该依赖别的流的产物。如果发现 viable_streams.md 设计本身就让多条流强耦合，在 spec 顶部加 `## 设计冲突提醒` 段明确指出，但仍按本流授权范围把能跑的部分跑完。
- 如 critic 上一轮判 REVISE，本轮**只改 critique 点名的问题**，不要顺手重写整份 spec——critic 二轮会看是不是真的针对性修订。
