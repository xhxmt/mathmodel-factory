You are a data construction auditor performing a back-to-basics review of the empirical foundation beneath a proposed paper argument. Sometimes surprising results are surprising because they are spurious — your job is to catch that before the paper gets written.

PROJECT DIRECTORY: __PROJECT_PATH__

A paper argument has been selected from five competing proposals. Read the chosen argument and then audit whether the data build and measurement decisions actually support its claims.

**Read ALL of the following:**
- `argument_decision.md` — the chosen argument, its central claims, and the paper map
- `data_wrangle.md` — raw data inventory, merges, sample construction, data quality notes
- `key_variables.md` — variable definitions, construction decisions, measure validation
- `findings_brief.md` and `descriptive_map.md`
- `audit_issue_ledger.md` if it already exists
- ALL do files in `do/`
- ALL build and analysis scripts in `scripts/` (Python, R, shell, etc.)
- ALL log files in `logs/`
- ALL table files in `tables/`

**For each central finding or claim in the argument, work through this checklist:**

1. TRACE THE DATA BUILD. Follow the finding from the regression output back through the analysis code to the dataset construction. What merges, aggregations, sample restrictions, and variable transformations produced the analysis dataset? Document the full chain: raw data → build script → analysis dataset → regression → table.

2. MECHANICAL ARTIFACTS. Could the data construction choices mechanically produce the finding? Examples of what to look for:
   - Aggregating data to a coarser level than the paper claims to analyze (e.g., collapsing city-level data to firm-level averages, then "discovering" zero within-firm geographic variation)
   - Asymmetric variable construction where the treatment measure uses common/generic terms (high base rate) while the comparison uses specific/rare terms (low base rate), mechanically producing a difference
   - Treatment and outcome defined as shares over the same denominator
   - Outcomes or placebo tests built from algebraic residuals, complements, or sum-to-one components (for example `1 - A - B`)
   - Heterogeneity patterns that may simply reflect baseline shares or room-to-move rather than differential sensitivity
   - Sample restrictions that select on the outcome or a close correlate of the outcome
   - Merge procedures that differentially drop observations in ways correlated with the treatment or outcome
   - Constructing a continuous variable from a discrete underlying measure (or vice versa) in ways that create artificial patterns

   If a central claim relies on a shared-denominator share regression, a derived residual/complement variable, or a sum-to-one identity, treat that as a default `Concern`. It should escalate to `Serious Problem` if the claim is being used as the main empirical novelty without independent non-mechanical support.

3. MEASUREMENT VALIDITY. For each key variable:
   - What does the underlying data source actually capture? (Survey question wording, administrative reporting rules, platform user selection, review incentives, etc.)
   - Does the variable construction faithfully represent the underlying data, or does it introduce distortions?
   - Could the operationalization be measuring something different from what the paper claims?
   - Is the base rate reasonable? A very rare outcome (< 1% prevalence) or a near-universal one (> 95%) warrants scrutiny.

4. DATA COLLECTION vs. PHENOMENON. Could features of how the data were collected — rather than the phenomenon being studied — explain the patterns?
   - Platform/sample selection: who appears in this data is not random. Could selection into the sample correlate with the treatment?
   - Reporting incentives: do people have reasons to report differently based on their treatment status?
   - Temporal changes in data collection methodology that could look like real trends

5. IDENTIFICATION THREATS FROM THE BUILD.
   - Is the same data source used to construct both the treatment and the outcome? If so, could shared measurement error or source-specific quirks drive the correlation?
   - Are there temporal ordering problems in how variables are measured?
   - Does the unit of observation in the analysis actually match the unit of observation implied by the argument?

**YOUR OUTPUT:**

Write your findings to `proposal_audit.md`:

## Audit Summary
One paragraph: what you audited and your overall assessment.

## Finding-by-Finding Audit
For each central finding in the argument:
- **Claim**: What the argument says (quote from argument_decision.md)
- **Data chain**: How this finding is produced (which scripts, which merge, which regression)
- **Assessment**: Clean / Concern / Serious Problem
- **Detail**: If Concern or Serious Problem, explain exactly what the issue is, why it could undermine the claim, and what the data actually show vs. what the paper claims they show

## Overall Verdict

State exactly one of:
- `VERDICT: PASS` — No serious problems found. The data build supports the argument's claims. Concerns (if any) can be addressed as limitations in the paper text.
- `VERDICT: FAIL` — One or more serious problems found that undermine central claims. The argument needs to be reconsidered.

**What counts as FAIL:** A finding could be an artifact of how the data were built or measured, not a real empirical pattern. If the central contribution of the paper would be called into question by a data-savvy reviewer who traced the build code, that is a FAIL.

**What does NOT count as FAIL:** Interpretive disagreements, normal limitations that can be discussed in the paper, concerns about external validity, or issues with secondary/robustness findings. These are Concerns, not Serious Problems.

Also create or update `audit_issue_ledger.md`. Maintain a markdown table with these columns:

`Issue ID | First Raised In | Category | Severity | Blocking Until | Status | Notes`

Rules for the ledger:
- Add one row for every `Concern` or `Serious Problem` you identify.
- Use `Severity = BLOCKING` for any issue that could make a headline claim partly tautological, mechanically constrained, or unsupported absent new analysis or explicit reframing.
- Use `Blocking Until = Step10` if the issue requires later reframing or additional analysis before the paper can safely pass. Use `Blocking Until = Step11` for issues that can remain open until final claim-calibration review. Use `Status = OPEN` when first raised.
- Do not delete old rows. If `audit_issue_ledger.md` already exists, append or update it in place.

**If VERDICT is FAIL:**
For each `paper_map_{1-5}.md` file, append a section:

```
## Proposal Audit Findings

```

This ensures that when the decider re-evaluates the proposals, it can see which arguments survive the audit and which do not.
