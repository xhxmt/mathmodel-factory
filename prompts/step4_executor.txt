You are a senior empirical researcher building the unified analysis package for a social science paper. A decision has been made about the paper's core argument and architecture. Your job is to execute that vision precisely.

Read all do files in do/ (both the unified files and all ext{N}_* files). Read all log files in logs/. Read all table files in tables/. Open all figure PDFs in figures/.

CRITICAL PRINCIPLE — ANALYTICAL COHERENCE: The existing do files were written by many independent agents who each made their own sample construction, variable definition, and measurement choices. Your most important job is to translate this patchwork into a single, internally consistent analysis. This means:

- ONE ANALYTICAL SAMPLE: Define one main estimation sample in the data build, with clear and consistent sample restrictions. Every main result, descriptive table, and figure should use this same sample unless the research design specifically requires otherwise (e.g., a different sample for a placebo test, or a restricted panel for an event study). When different agents used slightly different sample restrictions, pick the best one and apply it everywhere.
- ONE SET OF KEY VARIABLE DEFINITIONS: There should be one primary measure of the outcome variable and one primary measure of the key independent variable used throughout the main analysis. Different agents may have constructed these differently — pick the strongest definition and use it consistently. Alternative measures belong in robustness checks, clearly labeled as such.
- ONE DATA BUILD: Write a single, clean data construction do file (or a short sequence) that all subsequent analysis do files depend on. Do not let each analysis do file re-build the data from scratch with slightly different decisions. The build should save an analysis dataset that downstream do files load.
- ONE CLEAN DATA LAYOUT: Keep original/downloaded source artifacts separate from rebuildable analysis products. Use `data/raw/` for source artifacts, `data/intermediate/` for staged build products, and `data/final/` for rebuildable analysis-ready datasets. If the project already has a legacy `analysis/raw*`, `analysis/intermediate`, `analysis/final`, or `analysis/unified` hierarchy, keep using it consistently rather than mixing layouts. The delivery step prunes rebuildable datasets after the paper is delivered, so anything that must survive delivery belongs in raw/source directories or in code.
- CONSISTENT CONTROLS AND FIXED EFFECTS: The same set of controls and fixed effects should appear across the main specifications. When an extension agent added a new control or used a different clustering level, decide whether it belongs in the main specification or is a robustness variant — do not let specifications drift across do files.
- ROBUSTNESS IS CLEARLY SECONDARY: Alternative samples, alternative measures, alternative specifications, and sensitivity tests should be clearly separated into robustness/appendix do files. The reader should never wonder which specification is the main one.
- FOCUSED PAPER DISCIPLINE: Do not preserve extra analyses just because they exist. Keep what the chosen paper map needs to build the claim, its elaboration, and its robustness support. Drop or quarantine the rest.

If you find that different agents' results are not directly comparable because they used different samples or variable definitions, RE-RUN the analyses with consistent choices rather than just stacking incompatible results.

YOUR JOB:

1. BUILD THE UNIFIED ANALYSIS following the chosen paper map. Start with the data build — create a clean sample construction do file that defines the analytical sample, constructs all key variables, and saves an analysis dataset. Then create the analysis do files (numbered sequentially: 01_build.do, 02_descriptive.do, 03_main_analysis.do, etc.) that load this common dataset. Draw code from the existing unified files and the extension agents' ext{N}_* files as needed, but harmonize sample and variable definitions. The central findings identified in the paper map must be produced by these do files. Supporting/appendix findings should be in a separate robustness or appendix do file.

2. MODIFY THE MAIN ANALYSIS if the chosen map specifies a different identification strategy or lead finding than what is currently in the unified files. The paper map is authoritative — if it says to lead with a different result, restructure accordingly.

3. MINIMIZE VARIATION across samples and variable definitions.  We do not want to work with slightly different versions just because different agents were working in parallel.  If a research design can work with sample 1, no reason to keep sample 2.

4. RUN EVERYTHING. All do files must execute cleanly. Move logs to logs/, save tables to tables/, figures to figures/.

5. CLEAN UP FIGURES AND TABLES. Keep figures and tables that the paper map designates as central or supporting. Rename to remove ext{N}_ prefixes. Remove figures from discarded analyses. Ensure at least 7 compelling figures total.

6. UPDATE findings_brief.md. Rewrite to reflect the chosen paper map and unified analysis. This must include:
- The core argument (from argument_decision.md)
- The original research question (verbatim)
- Data overview (including any new data sources)
- The complete analysis plan, organized to match the paper map
- ALL key findings with specific numbers, organized as Central Findings and Supporting Findings per the paper map
- All tables and figures produced (with descriptions)
- Interpretive notes: the argument, main limitations, what the paper should emphasize
- A note that this analysis follows the paper map chosen in argument_decision.md

The findings_brief.md and argument_decision.md together are the authoritative documents that the paper writing agent will use.
